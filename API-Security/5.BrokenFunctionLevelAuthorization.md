# Broken Function-Level Authorization

Broken function-level authorization occurs when an application fails to properly enforce access controls on specific functions or features, allowing unauthorized users to access functionality they should not have permission to use. This vulnerability can lead to various security consequences and requires robust prevention strategies.

## Consequences:

1. **Unauthorized Access**: Attackers can exploit broken function-level authorization to gain access to restricted features or data within the application, potentially compromising sensitive information.

2. **Privilege Escalation**: Users might be able to escalate their privileges within the application, gaining access to administrative functions or other users' accounts.

3. **Data Manipulation**: Attackers could modify or delete critical data within the application if they gain access to functions they shouldn't have permission to use.

4. **Increased Risk of Forced Browsing**: Broken function-level authorization may exacerbate the risk of forced browsing attacks, allowing attackers to access additional resources and directories not properly protected by access controls.

5. **Regulatory Compliance Violations**: Failure to properly implement function-level authorization controls can lead to violations of regulatory requirements regarding data protection and privacy.

## Prevention Strategies:

1. **Implement Proper Access Controls**: Ensure that access controls are implemented at the function level, restricting access to features and data based on user roles and permissions.

2. **Validate User Permissions**: Before allowing access to a specific function, verify that the user has the necessary permissions to perform that action.

3. **Use Role-Based Access Control (RBAC)**: Utilize RBAC to manage user permissions effectively, assigning roles to users and defining which functions they can access based on those roles.

4. **Implement Strong Authentication**: Enforce strong authentication mechanisms to verify the identity of users and prevent unauthorized access.

5. **Protect Against Forced Browsing**: Extend access controls beyond application functions to encompass all resources accessible via the application, including directories and files susceptible to forced browsing attacks.

6. **Regular Security Audits**: Conduct regular security audits and testing to identify and address any vulnerabilities related to function-level authorization and forced browsing.

By understanding the consequences of broken function-level authorization and implementing comprehensive prevention strategies, developers can enhance the security of their applications and mitigate the risk of unauthorized access and data exposure.


# Examples

## Claims Processing System

Let's consider a claims processing system for an insurance company as our example. This system manages insurance claims submitted by policyholders and processes them according to predefined rules and workflows. In this scenario, we can illustrate Broken Object Level Authorization, Broken Object Property Level Authorization, and Broken Function Level Authorization as follows:

1. **Broken Object Level Authorization**:
    - **Definition**: Broken object-level authorization occurs when the system fails to properly restrict access to entire objects or resources based on user roles or permissions.
    - **Example**: In our claims processing system, broken object-level authorization might occur if a claims adjuster, who should only have access to view and process claims assigned to them, is able to view and modify claims assigned to other adjusters.


2. **Broken Object Property Level Authorization**:
    - **Definition**: Broken object property-level authorization occurs when the system fails to properly restrict access to specific properties or attributes of an object based on user roles or permissions.
    - **Example**: In our claims processing system, broken object property-level authorization might occur if a claims adjuster, who should only have permission to view and update certain fields of a claim (e.g., status, notes), is able to modify sensitive fields such as payout amounts or policy coverage details.


3. **Broken Function Level Authorization**:
    - **Definition**: Broken function-level authorization occurs when the system fails to properly enforce access controls on specific functions or features within the application.
    - **Example**: In our claims processing system, broken function-level authorization might occur if a user without administrative privileges is able to access and execute administrative functions, such as deleting claims or modifying system settings.


In summary, while broken object-level, object property-level, and function-level authorization address different aspects of access control within an application, they can overlap in scenarios where the failure to enforce proper access controls impacts access to entire objects, specific properties within those objects, or individual functions and features. However, each vulnerability also has its distinct focus and implications for security.

# Online Banking System Example

Let's consider an online banking system as another example. This system allows users to perform various banking operations such as checking account balances, transferring funds, and paying bills. We can illustrate Broken Object Level Authorization, Broken Object Property Level Authorization, and Broken Function Level Authorization in this context:

1. **Broken Object Level Authorization**:
    - **Definition**: In the context of the online banking system, broken object-level authorization occurs when the system fails to properly restrict access to entire banking objects or resources based on user roles or permissions.
    - **Example**: Suppose a regular user should only have access to view their own account details and transactions. However, due to a vulnerability, they can access the account details of other users or even administrative functions meant for bank employees.

2. **Broken Object Property Level Authorization**:
    - **Definition**: Broken object property-level authorization occurs when the system fails to properly restrict access to specific properties or attributes of an object based on user roles or permissions.
    - **Example**: Continuing with the online banking system, broken object property-level authorization might occur if a user is able to view or modify sensitive account details such as account balance, transaction history, or personal information of other users.

3. **Broken Function Level Authorization**:
    - **Definition**: Broken function-level authorization occurs when the system fails to properly enforce access controls on specific functions or features within the application.
    - **Example**: In the online banking system, broken function-level authorization might occur if a regular user can access administrative functions such as adding or removing users, adjusting account privileges, or altering system configurations.

In summary, using the online banking system example, we can understand how broken object-level, object property-level, and function-level authorization vulnerabilities manifest and their implications for security within the application. Each vulnerability addresses different aspects of access control, but they can overlap in scenarios where the failure to enforce proper access controls impacts access to entire objects, specific properties within those objects, or individual functions and features.
